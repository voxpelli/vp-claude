## Bugs

- **Prompt hooks can't call MCP tools** (2026-03-14, refined 2026-04-29) \[degraded\] ‚Äî `type: "prompt"` hooks spawn a separate Haiku instance with no MCP tool access. Any prompt hook that references MCP tools (e.g., `mcp__basic-memory__edit_note`) is silently non-functional ‚Äî it runs without error but can't execute the tools. This isn't documented in the hook type reference; the workaround is to use `type: "command"` with `additionalContext` to inject instructions into the main session which retains full MCP access. **Refinement (Sprint 18):** the bug is scoped to `type: "prompt"`. `type: "mcp_tool"` (Claude Code v2.1.118+) IS a first-class handler that invokes MCP tools directly with no shell subprocess and no Haiku dispatch ‚Äî the prompt-hooks restriction does NOT apply to it. Cost us 10 versions of silent PreCompact failure (v0.1.0‚Äìv0.10.1, hook retired in v0.28.0).

- **Hook stdout reads only first JSON object** (2026-03-28) \[degraded\] ‚Äî When a command hook script emits multiple JSON objects on stdout (e.g. separate `cat <<'EOF'` blocks for `systemMessage` and `additionalContext`), Claude Code parses only the first object and silently drops the rest. This is documented but non-obvious. Workaround: merge all output into a single JSON object using `jq -n`. Cost us silent loss of priming suggestion + audit reminder in session-start.sh since v0.15.0.

- **PostToolUseFailure shows misleading "hook stopped continuation" label** (2026-03-28) \[cosmetic\] ‚Äî When a PostToolUseFailure prompt hook fires, the Claude Code framework wraps the output with a "hook stopped continuation" label in the TUI. This is misleading ‚Äî the hook didn't stop anything, the tool itself failed before the hook ran. Cannot fix from plugin side. Tracked: claude-code#27886, #18427, #17088.

- **Embedded `grep`‚Üí`ugrep` shim amplifies regex memory** (2026-07-01) \[degraded\] ‚Äî Claude Code's Bash-tool `grep` shim resolves to `exec -a ugrep "$CLAUDE_CODE_EXECPATH" ...` ‚Äî i.e. `grep` calls run inside the full `claude` binary, not a lean standalone `ugrep`. The binary is a **Bun v1.4.0 / JavaScriptCore** build (verified directly via `strings`/`otool` ‚Äî Mach-O sections `__jsc_int`/`__jsc_opcodes`/`__bun`; no V8), and it embeds what appears to be the real Genivia `ugrep 7.5.0`/RE-flex engine (a genuine `ugrep 7.5.0` version banner + Genivia/RE-flex attribution text are present in the binary) rather than a from-scratch JS reimplementation ‚Äî so the upstream issue's "V8-heap" and "catastrophic backtracking" framing (RE-flex is DFA-based, not a classic backtracking engine) are both likely imprecise, though the exact allocation path wasn't pinned down further (no disassembly). Piping `strings -a` over the 227 MB `claude` binary itself (which embeds a ~467,000-character single-line minified JS blob) into `grep -o '.{0,80}X.{0,120}'` ballooned this wrapper to a **14 GB** phys_footprint peak, independently caught by a sibling local memory-monitoring toolkit, before self-terminating. Reported (not independently verified) comparison: ~754 MB for a real standalone `ugrep` and ~2.8 MB for real `/usr/bin/grep` on an equivalent repro. Independently reproduced by others on macOS, WSL2, and bare-metal Linux through at least v2.1.183. \[upstream: [anthropics/claude-code#54394](https://github.com/anthropics/claude-code/issues/54394)\]
  Severity: degraded ¬∑ Ownership: upstream ¬∑ Workaround: partial ‚Äî use `command grep` or an explicit path (`/usr/bin/grep`), or `rg`, to bypass the shim; fold long lines (`fold -w 2000`) before matching when scanning binaries with large embedded string blobs.

- **Bash-tool timeout doesn't kill the full child process group ‚Äî detached children keep running** (2026-07-01) \[degraded\] ‚Äî A subagent's Bash-tool call (a multi-command `strings | grep` shell block) returned/timed out, but its spawned shell and pipeline children kept running **detached** for roughly 13 more minutes ‚Äî invisible to any PID-tree-based orphan/leak detector, since the parent shell was never reparented to PID 1; the process tree stayed fully intact, only the _logical_ owner (the tool call itself) had ended. Distinct root cause from the PID-1-reparenting orphan cluster (#1935/#27707/#33947, tracked separately in a sibling toolkit's roadmap) ‚Äî this is about process-group kill on timeout, not reparenting. \[upstream: [anthropics/claude-code#68647](https://github.com/anthropics/claude-code/issues/68647)\]
  Severity: degraded ¬∑ Ownership: upstream ¬∑ Workaround: none ‚Äî no user-facing way to detect or terminate a detached tool-call child from within Claude Code; it relies on the command finishing or exhausting on its own.
